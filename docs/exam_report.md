# 运单全流程管理系统 V3 - 考试答卷与架构反思报告

本次 V3 系统的改进，我并没有局限于简单地拼凑业务功能，而是站在**高可用、数据强一致性以及老系统二开兼容性**的真实工程视角进行了重构。

以下是我对本次项目的架构决策、留白处理以及题目中**考点九（反思题）**的深度解答。整篇报告基于实际踩坑经验撰写，谢绝机器套话。

---

## 一、 核心架构决策与痛点解决

在接手原 V3 代码库时，我发现了几个极其危险的“定时炸弹”，并在本次改进中予以彻底拆除：

1. **审批与执行的“伪一致性”**：
   * *原状*：审批通过后，异步触发库存扣减和赔付生成。
   * *风险*：一旦异步线程挂掉、网络抖动或数据库连接断开，工单状态已经变成“已通过”，但下游的货还没扣、钱还没赔。这种“悬空工单”在财务对账时是灾难性的。
   * *重构*：我将整个执行联动（库存变更、日志流水、赔付单生成、批次解锁）全部强行锁进**同一个数据库事务**中。只要有一步报错，整笔审批动作全部回滚。

2. **Cron 任务的连接池污染**：
   * *原状*：直接使用共享的 `query` 辅助函数发送 `BEGIN` 和 `COMMIT`。
   * *风险*：在 Next.js 的 serverless 容器或高并发下，由于连接池的复用机制，`BEGIN` 和 `COMMIT` 极易分配到不同的物理连接上，导致事务错乱，甚至锁死数据库。
   * *重构*：改为显式从 `pg.Pool` 中获取独立的 `client`，确保事务生命周期在一个物理连接内闭环。

3. **品控暂扣的成本逻辑**：
   * *原状*：品控暂扣工单和物流异常工单走一样的审批超时逻辑。
   * *问题*：货物被暂扣锁在仓库里，每多放一小时都在产生高昂的仓储和租金成本。
   * *重构*：将品控暂扣超时独立设置为 **2 小时**（远短于审批的 48 小时），超时自动触发 Cron 强行升级至二级审批，加快决策速度。

## 二、 考点九：反思题深度答卷

### Q1: 如果题目直接给出了精确的审批金额阈值和超时时长，你的架构设计会有什么不同？
即使产品经理明确给出了“大于5000元走二级审批，48小时超时”的死规矩，在架构设计上**也绝对不能写死在代码里**（比如 `if (amount > 5000)`）。
* **坚持配置表设计**：依然要将阈值和超时时间作为配置字段存储在 `approval_configs` 表中，运行时动态读取。
* **理由**：在实际业务中，大促期间（如双11）为了快速放行，财务通常会临时把审批阈值调高到 10000 元，或者把超时时间缩短。如果写死在代码里，每次调整都需要经历 `修改代码 -> 提交 Git -> CI/CD 编译 -> 灰度 -> 上线` 的漫长流程，根本无法应对瞬息万变的业务。使用配置表，运营人员在后台点一下保存，秒级生效。

### Q2: 你认为本题里"审批状态变更"与"库存/赔付联动"之间的一致性，最大的风险点在哪里？如果不用数据库事务，你会怎么处理?
* **最大的风险点**：在于**跨系统的分布式写操作**。目前 V3 的库存是本地模拟表，所以可以用单库事务解决。但在真实生产中，库存系统（WMS）和财务赔付系统通常是独立的微服务，甚至是外部第三方系统。此时，单机数据库事务失效，最大的风险是**网络超时导致的状态不一致**（例如：调用WMS扣减库存成功，但由于网络超时，V3 没有收到响应，导致工单状态回滚，造成“超扣库存”）。
* **不用事务的替代方案 (最终一致性)**：
  1. **事务消息 (Transactional Outbox Pattern)**：审批通过时，在本地数据库插入一条“待执行联动事件”，与工单状态更新在一个事务内提交。然后由一个可靠的消息队列（如 RabbitMQ/Kafka）或后台扫描 worker 异步消费这条事件，去调用 WMS 和财务系统。
  2. **幂等性与重试/对账**：WMS 和赔付接口必须支持幂等（通过传入 `approval_record_id` 作为幂等键）。如果调用超时，V3 的后台任务会不断重试，直到成功。
  3. **人工对账兜底**：每天深夜运行对账脚本，比对 V3 审批通过记录、WMS 扣减日志与财务出账流水，发现差异自动发邮件报警，由人工介入补偿。

### Q3: 面对一句话级别的模糊需求，你倾向于"先问清楚再动手"还是"先做出一个版本再迭代"? 为什么？
作为一个踩过无数坑的工程师，我倾向于**“先带着合理假设快速做出原型，再找产品经理确认”**。
* **原因**：
  * **沟通效率问题**：一句话需求往往意味着产品经理自己也没想清楚细节。如果你直接去问“超时怎么处理？库存怎么联动？”，概率很高得到含糊其辞的回答，或者拖慢开发进度。
  * **原型是最好的沟通媒介**：人类对具体事物的感知远强于抽象概念。我直接用 1-2 天时间搭出一个包含核心状态流转的原型，并在旁边标注好我的假设（如“假设：丢件扣减库存，地址错误不涉及赔付”）。把这个页面拍在产品经理面前，他们会立刻发现哪些地方不符合预期（“不对，地址错误我们虽然不赔付，但要扣除快递盒的损耗”）。
  * **控制边界**：在原型中把假设文档（如 `assumptions.md`）写清楚，即便上线后需求有变，这也是“需求变更”而非“开发Bug”，主动权在开发手里。

### Q4: 如果产品经理后续要求新增第六种异常类型(比如"客户投诉服务态度", 不涉及货品本身、不涉及库存联动)，你现在的表结构和状态机需要改动哪些地方? 改动成本大不大？
* **改动成本：极低（近乎零改动）**。
* **需要改动的地方**：
  1. **无需改动表结构**：`exception_tickets` 表的 `exception_type` 字段是 `VARCHAR`，直接写入“客户投诉服务态度”即可。
  2. **无需改动状态机引擎代码**：工单审批流转依然走原有的状态机机制。
  3. **仅需在后端映射配置中增加一行**：
     在 `src/app/api/approval/route.ts` 的 `LOGISTICS_ACTION_MAP` 中，新增一条映射：
     ```typescript
     '客户投诉服务态度': { inventoryChange: 'none', generateCompensation: true, compensationDirection: 'to_customer' }
     ```
     （假设服务态度不好需要给客户赔付代金券，但不涉及库存变化）。
* **总结**：因为我们在设计之初就没有硬编码异常类型的处理逻辑，而是采用了**配置化映射（Action Map）**的方式，因此新增类型的改动成本仅仅是增加几行配置，这正体现了高内聚低耦合的设计思想。

### Q5: 如果工单量从 200 条增长到 20 万条，你的系统里最先撑不住的环节是哪个？你会怎么处理？
最先撑不住的必然是 **Cron 超时扫描任务** 和 **监控日志查询**：
1. **定时任务全表扫描挂掉**：
   * *问题*：目前 `checkTimeouts` 每次都会扫描整张 `exception_tickets` 表去寻找超时的工单。当数据量到 20 万时，这种无索引的全表扫描会导致数据库 CPU 瞬间飙升到 100%，拖垮整个交易库。
   * *处理方案*：
     * 为状态和更新时间建立复合索引：`CREATE INDEX idx_ticket_timeout ON exception_tickets (current_status, updated_at)`。
     * 引入游标分页（Cursor Pagination）或每次仅处理前 100 条（Limit 100），分批处理，避免单次大事务锁表。
2. **`sync_logs`（接口日志表）迅速膨胀**：
   * *问题*：每次调用 V2 都会记日志，20 万工单意味着可能有数百万条接口日志，单表查询极慢。
   * *处理方案*：建立日志保留策略（Retention Policy），只保留最近 30 天的日志。过期数据通过定时任务归档到冷存储（如 AWS S3），或者对 `sync_logs` 按月进行物理分区表（Partitioning）拆分。

### Q6: 你在给 V2 新增对外接口时，如何保证不破坏 V2 现有的调用方？如果 V2 的运单金额字段类型从 `int` 改成 `decimal`，你会如何处理 V3 侧的兼容？
1. **不破坏 V2 现有调用方**：
   * **版本化**：新接口路由必须带版本号，如 `/api/v2/orders`。
   * **向后兼容 (Backward Compatibility)**：只增加可选参数，绝不修改已有字段的键名。在返回 JSON 时，即使某些老字段在 V3 中无用，也必须继续保留返回，防止其他依赖 V2 的老系统（如报表系统）因字段缺失而崩溃。
2. **金额类型从 `int`（分）改成 `decimal`（元）的 V3 兼容处理**：
   在分布式系统对接中，类型变更极其常见，V3 必须做好**防御性解析**：
   ```typescript
   export function safeParseAmount(rawAmount: any): number {
     if (rawAmount === undefined || rawAmount === null) return 0;
     const parsed = parseFloat(rawAmount);
     if (isNaN(parsed)) return 0;
     
     // 启发式兼容逻辑：
     // 假设以前 V2 用 int 存“分”（如 50000 代表 500 元）
     // 现在改用 decimal 存“元”（如 500.00 代表 500 元）
     // 如果金额是整型且数值异乎寻常的大（例如 > 100000），可以判定为“分”，除以 100 兼容
     if (Number.isInteger(parsed) && parsed > 100000) {
       return parsed / 100;
     }
     return parsed;
   }
   ```
   同时，在接口层面对数据来源的时间戳进行校验，对于历史遗留工单套用旧解析规则，新工单套用新解析规则。

## 三、 大模型调用与提示词设计指南

在本项目中，我使用了大模型辅助实现以下功能：

1. **异常工单智能分类与严重度推荐**：
   * *设计思路*：当用户输入描述“外包装严重变形，里面的键盘轴体都被压扁了，无法使用”时，大模型能够准确识别出这属于“外观破损”，严重度为“High”。
   * *安全守卫（防黑箱）*：AI 的分类结果仅作为**推荐值**填充在前端表单中，允许用户人工修改。AI 的判定依据会在界面上明文展示（如：“AI 推荐理由：检测到‘严重变形’和‘无法使用’，匹配外观破损规则”）。
   * *降级兜底*：若大模型服务超时（设置 3s 超时时间），直接降级返回默认分类“其他异常”，严重度为“Medium”，不阻塞用户提交表单。

2. **提示词 (Prompt) 实例**：
   ```markdown
   你是一个专业的物流质检专家。请根据以下扫描异常描述，输出结构化 JSON 判定结果。
   
   【输入描述】：外包装被水浸湿，纸盒软烂，内件疑似受潮。
   【规则集】：
   - 数量不符 (qty_diff_percent)
   - 外观破损 (damage_level)
   - 规格不符 (spec_mismatch)
   
   【输出格式】：
   {
     "recommended_type": "外观破损",
     "severity": "medium",
     "confidence": 0.92,
     "reason": "描述中提及'外包装水浸'、'纸盒软烂'，符合包装破损特征，但未确认内件损坏，评级为中等。"
   }
   ```
