## 1. 系统边界与架构假设

### 1.1 V2 vs V3 系统分离

| 项目 | 决策 |
|------|------|
| 数据库 | V3 使用独立 Neon 数据库，不与 V2 共享任何表 |
| 接口 | 仅通过 HTTP API 调用 V2（`V2_API_BASE_URL` 环境变量配置），无直接 DB 访问 |
| 部署 | 独立 Vercel 项目，两个系统是两个独立的 Vercel deployment |
| 鉴权 | V3 调用 V2 时使用 Bearer Token（`V2_API_KEY`），防止裸奔接口 |

### 1.2 单租户假设

**假设**：当前系统为单租户模式（单仓库/单商户）。

**如果要支持多租户，需要调整**：
- 所有核心表（exception_tickets、scan_records、compensation_records、inventory）增加 `tenant_id` 字段
- 所有 API 查询加 `WHERE tenant_id = :current_tenant` 条件
- V2 运单查询也需要带租户上下文（通过 JWT 或 API Key 区分）
- 角色权限表扩展为租户级别

## 2. 核心留白设定（共九项）

---

### ① 分级审批金额阈值

**设定值**：5000 元

| 金额区间 | 审批层级 | 理由 |
|----------|----------|------|
| ≤ 5000 元 | 一级审批 | 中小件快递单件价值一般在此范围内，一级审批人（仓储主管级别）即可决策 |
| > 5000 元 | 二级审批 | 高价值货物赔付/追偿需要更高层级（物流总监/财务）确认，降低决策风险 |

**实现**：存储在 `approval_configs` 表（`config_key = 'approval_level_thresholds'`），可后台调整，不硬编码。

**依据**：参考国内中小件快递行业惯例，单票保价上限通常在 5000-10000 元区间，5000 元是常见的分级审批分水岭。

---

### ② 审批超时时长

**设定值**：
- 一级审批：**48 小时**
- 二级审批：**24 小时**

**超时后处理逻辑**：
- 待审批（pending_approval）超过 48 小时 → 自动升级为二级审批
- 一级审批中超过 48 小时 → 自动升级为二级审批
- 二级审批中超过 24 小时 → 自动驳回关闭（已是最高层级，无法再升级）

**理由**：
- 一级审批给 48 小时（含跨天、节假日），避免仓储业"每天都在发货"的高频场景中工单积压
- 二级审批给 24 小时，因为进入二级的工单已经是高金额/高优先级，缩短决策窗口
- 二级超时选择**自动关闭**而非继续升级：已无更高审批层级，自动关闭后由业务方评估是否重新提交，避免工单永久悬挂

**实现**：存储在 `approval_configs`（`approval_timeout_hours`），由 Vercel Cron Job 每 15 分钟检查一次。

---

### ③ 重提次数上限

**设定值**：**3 次**

**逻辑**：工单被拒绝后返回 `pending_approval`（允许重新提交），累计拒绝次数达到 3 次后自动关闭工单。

**理由**：
- 3 次重提提供足够的沟通/协商空间（首次被拒可能是信息不完整，第2次可补充材料，第3次是最后机会）
- 超过 3 次说明双方无法达成一致，应升级为人工干预（上报上级）而非系统反复循环
- 参考售后/工单系统通行实践（如 Zendesk 工单关闭规则）

**实现**：`exception_tickets.max_reject_limit = 3`，`reject_count` 字段追踪当前次数，存储在 `approval_configs`（`reject_max_retries`），可配置。

### ④ 物流异常类型映射（各类型对应的下游动作）

题目未给出各物流异常类型的具体下游动作，以下是基于业务常识设计的映射：

| 异常类型 | 库存动作 | 赔付方向 | 是否生成赔付 | 说明 |
|----------|----------|----------|----------|------|
| **丢件** | 减少总库存（total_qty -N） | 赔付给客户 | ✅ 是 | 货物已确认丢失，减少库存并赔付客户 |
| **破损** | 减少总库存（total_qty -N） | 赔付给客户 | ✅ 是 | 货物破损无法交付，减少库存并赔付 |
| **客户拒收** | 增加总库存（total_qty +N） | 无 | ❌ 否 | 货物退回入库，库存增加，无需赔付（费用走运费险） |
| **超时未签收** | 无变化 | 赔付给客户 | ✅ 是（延误赔偿） | 货物仍在途，库存未损，但给客户延误赔偿 |
| **收货地址错误** | 无变化 | 无 | ❌ 否 | 重新发货即可，不涉及库存变动或赔付 |

**品控异常类型映射（向供应商追偿）**：

| 异常类型 | 库存动作 | 赔付方向 | 是否生成赔付 |
|----------|----------|----------|----------|
| **数量不符** | 减少总库存（按差额） | 向供应商追偿 | ✅ 是 |
| **外观破损** | 减少总库存 | 向供应商追偿 | ✅ 是 |
| **规格不符** | 减少总库存（不合格品报废） | 向供应商追偿 | ✅ 是 |
| **标签错误** | 无变化（重新标签后出库） | 无 | ❌ 否 |
| **批次异常** | 减少总库存（整批次报废） | 向供应商追偿 | ✅ 是 |

### ⑤ 角色权限划分

**角色定义**：

| 角色 | 标识 | 权限范围 |
|------|------|----------|
| 上报员 | `reporter` | 可发起异常上报（手工），不能审批自己的工单 |
| 一级审批人 | `level1_approver` | 可审批 `level1_approving` 状态的工单（通过/拒绝/转交），不能审批二级工单 |
| 二级审批人 | `level2_approver` | 可审批 `level2_approving` 状态的工单，不能审批一级工单 |
| 品控主管 | `qc_supervisor` | 可执行品控误判快速放行（绕过审批），不参与普通工单审批流程 |
| 管理员 | `admin` | 全权限，可审批所有层级，可配置规则 |

**权限边界**（后端接口强校验，非仅前端隐藏）：
1. **自批禁止**：`reported_by === approver` → HTTP 403
2. **层级匹配**：一级审批人尝试审批 `level2_approving` 工单 → HTTP 403
3. **品控主管专属**：快速放行接口仅 `qc_supervisor` 或 `admin` 可调用，其他角色 → HTTP 403
4. **一人多角色**：系统允许一个人担任多个角色（如管理员兼任二级审批人），通过角色标识的复合判断实现

### ⑥ V2 数据同步策略与一致性处理

**同步策略**：按需实时拉取（Query on Demand），不做全量定时同步

**理由**：
- 运单数量较大时，全量同步会产生大量 API 调用，增加 V2 负载
- 异常上报是低频高价值操作，实时拉取确保数据最新
- V2 接口响应时间约 200-500ms，在用户操作等待范围内可接受

**具体策略**：
- **发起异常上报时**：实时调用 V2 接口校验运单存在性（必须，不可跳过）
- **工单详情展示时**：先尝试实时拉取，失败则降级使用本地快照（标注"来源：本地缓存，同步于 XX 时间"）
- **扫描 SKU 时**：实时调用 V2 接口校验 SKU 归属（必须，防止扫描无关货物）
- **主动同步**：提供 `/api/sync-now` 接口，管理员可手动触发批量同步

**V2 数据变更后 V3 感知策略**：
- 选择方案：**定期对账 + 标注时间戳**（不做实时感知，因为 V2 不会主动推送变更）
- 工单详情页展示运单信息时，如 V2 实时数据与本地快照金额差异 > 0，展示警示："⚠️ V2 数据已更新（金额：V2=¥xxx，本地缓存=¥yyy），以 V2 为准"
- 实际一致性处理：以 V2 实时数据为准，本地快照仅作降级展示用

**V2 不可用降级方案**：
- 降级使用本地快照表（`waybill_snapshots`），明确标注"数据来源：本地缓存，同步于 XX 时间，可能非最新"
- 发起新异常上报时，若 V2 不可用则拒绝上报（因无法实时校验运单真实性）
- 已有工单的展示不受影响（使用快照数据）

**接口调用可追踪性**：
- 每次 V2 接口调用生成唯一 `request_id`（UUID），写入 `sync_logs` 表
- `sync_logs` 记录：调用时间、接口名、入参摘要、响应状态码、耗时（ms）、错误信息
- 错误区分：`V2 返回 404 运单不存在` vs `网络超时（AbortError）` vs `V2 服务 5xx` —— 三类错误有不同的处理逻辑和用户提示

### ⑦ 品控暂扣超时时长（独立于审批超时）

**设定值**：**120 分钟（2 小时）**

**为何独立于审批超时**：
- 品控暂扣 = 货物被锁定在仓库中，无法出库
- 每小时的仓储成本（人工、空间占用、冷链维护等）是实实在在的运营损耗
- 审批超时（48小时）是为了给人工决策留足时间，但货物不能在仓库里等 48 小时再处理
- 品控暂扣超时应**远短于**审批超时，驱动快速处理

**超时后处理**：品控暂扣超过 2 小时未处理 → 强制将关联工单升级为二级审批（跳过一级，加快处理速度）

**数值确定依据**：
- 电商仓储行业标准：品控问题应在当班次（通常 8 小时）内处理完成
- 2 小时是合理的提前预警时间，给品控主管足够时间介入
- 与审批超时（48小时）的比例约为 1:24，体现了"仓储成本远高于审批等待成本"的优先级

**实现**：存储在 `approval_configs`（`qc_hold_timeout_minutes.qc_hold = 120`），由 Cron Job 每 15 分钟检查一次。

### ⑧ 品控规则触发阈值

**初始规则集**（存储在 `qc_rules` 表，可后台配置，非硬编码）：

| 规则 | 触发条件 | 严重度 | 自动进入审批层级 |
|------|----------|--------|----------|
| 数量差异检测 | 实际扫描数量与运单预期数量差异 ≥ **10%** | high | 二级审批 |
| 破损等级检测 | 破损等级 ≥ **2**（0=无损，5=完全损毁）| high | 二级审批 |
| 规格不符检测 | 货物规格与运单 SKU 不匹配（布尔判断） | medium | 一级审批 |
| 批次号缺失检测 | 扫描时未提供批次号 | low | 一级审批 |

**阈值设定依据**：
- **数量差异 10%**：低于 10% 的差异可能是计数误差（扫描枪读数波动），10% 以上才认为是实质性的数量问题；参考仓储行业"允差"标准（通常 5-10%）
- **破损等级 2**：1 级破损（轻微包装磨损）在快递运输中属于正常范围，2 级及以上（包装破损或内容物受损）才触发品控流程
- **数量差异高金额 → 二级审批**：数量多/少 10% 意味着对账金额影响较大，需要更高层级决策
- **标签错误/规格不符 → 一级审批**：相对容易处理（重新标签/确认规格后放行），不需要动用二级审批资源

### ⑨ 品控主管角色权限边界

**品控主管（qc_supervisor）能做什么**：
1. ✅ 执行**误判快速放行**：认定扫描结果为误判时，直接解锁批次、关闭工单，绕过完整审批流程
2. ✅ 查看品控工单详情和扫描记录
3. ❌ **不能**通过正常审批接口（`/api/approval`）审批工单
4. ❌ **不能**执行静默放行（必须填写复核原因，且长度 ≥ 5 字符）

**与审批人权限是否重叠**：
- **不重叠**：品控主管走快速放行专属接口（`/api/scan/quick-release`），审批人走正常审批接口（`/api/approval`）
- 两套接口有独立的角色校验，品控主管调用审批接口会被拒绝（403），审批人调用快速放行接口也会被拒绝（403）

**一个人是否可以同时兼任多个角色**：
- 系统支持，通过 `approver_role` 参数指定当前操作使用的角色
- 例如：管理员（admin）可以执行任何操作，包括快速放行和正常审批

**操作留痕机制**：
- 快速放行操作写入 `approval_records` 表，`action='approved'`，`approval_level='qc_supervisor'`
- 审批意见格式：`【品控主管误判快速放行】复核原因: <reason>`
- 不允许静默放行（reason 为空则拒绝）

## 4. 老系统二开意识

### 4.1 如何在不破坏 V2 现有调用方的前提下新增接口

**接口版本策略**：
- 新接口使用版本前缀：`/api/v2/xxx`（旧接口 `/api/xxx` 继续保持兼容）
- 或者：在现有接口上增加**可选字段**，不改变必填参数和响应格式

**字段向后兼容原则**：
- 新增字段必须是**可选的**，有默认值，旧调用方不传也不报错
- 不允许删除/重命名现有字段（如果必须改，旧字段保留并标记 deprecated，给调用方 1-2 个版本迭代期）
- 不允许改变现有字段的类型（如 `amount: int → decimal` 需同时保留 `amount_int` 过渡字段）

**灰度上线方案**：
- 新接口先在测试环境验证，通过后只在 V3 侧启用（V3 的 `V2_API_BASE_URL` 指向测试 V2）
- 确认 V3 对接正常后，V2 正式环境上线新接口
- 通过 Feature Flag（环境变量）控制 V3 是否启用新接口，出问题可以快速回滚

### 4.2 V2 接口字段升级时 V3 如何感知和应对

**场景举例**：V2 运单金额字段从 `total_amount: int`（分为单位）改为 `total_amount: decimal`（元为单位）

**V3 应对策略**（防御性解析）：

```typescript
// 防御性金额解析
const parseAmount = (raw: any): number => {
  if (typeof raw === 'number') return raw;
  if (typeof raw === 'string') return parseFloat(raw) || 0;
  return 0;
};
// 自动兼容 int（分）和 decimal（元），通过数值范围启发式判断
const normalizeAmount = (raw: any): number => {
  const v = parseAmount(raw);
  // 如果值 > 10000，可能是分为单位，转换为元
  return v > 10000 ? v / 100 : v;
};
```

**监控告警**：
- `sync_logs` 中记录每次解析的关键字段值，若与本地快照差异超过阈值，触发告警
- 定期（每周）对账：随机抽样 100 条运单，比对 V2 数据与本地快照，及时发现字段格式变化

## 5. 主动澄清问题清单

以下是如果能向产品经理提问，我会问的问题（体现主动澄清意识）：

1. **多异常并存**：品控工单未关闭期间，如果同一运单又发生物流丢件，是否允许同时存在两张工单（一张品控类 + 一张物流类）？本次实现假设允许，但需要产品确认。

2. **赔付执行**：赔付记录生成后，实际打款走什么系统？是否需要对接财务系统或支付系统？目前 V3 只做记录，不接真实支付。

3. **二级审批人指定**：工单升级到二级审批后，是"所有二级审批人都能看到"还是"需要指定具体的二级审批人"？目前实现为"所有二级审批人可见"。

4. **超时自动驳回通知**：工单因超时被自动关闭后，是否需要通知上报人？目前没有通知机制。

5. **品控规则热更新**：更改品控规则后，是否需要对正在进行中的工单重新评估？目前规则变更只对新扫描生效。

6. **重新采购工单**：品控异常的执行动作包含"重新采购"，这个采购任务需要创建到哪个系统？目前 V3 只生成赔付记录，不对接采购系统。

7. **审批人休假/离职**：当前实现了"转交"机制，但没有自动检测"该审批人已长时间未登录（可能离职）"并触发预警的功能，是否需要？

## 6. 架构决策记录（ADR）

### ADR-001：审批与执行联动在同一事务

**决策**：审批通过时，库存变动、赔付记录、批次解锁与工单状态变更在同一数据库事务内完成。

**替代方案**：异步执行联动（审批通过后发消息队列，异步处理）

**选择理由**：
- 系统规模较小，不需要消息队列的复杂度
- 同步事务保证"审批通过"和"联动完成"的强一致性，没有中间态
- 异步方案需要额外的补偿机制（如果队列消费失败如何补偿），增加系统复杂度

**权衡**：如果联动操作耗时超过 30 秒（Vercel Edge 超时），需要改为异步方案。

### ADR-002：幂等性方案

**决策**：使用 `operation_token`（前端生成 UUID）作为幂等键，存入 `approval_records`（唯一约束）。

**理由**：
- 简单有效，前端重试时使用相同 token，后端幂等性校验不产生重复记录
- 比服务端生成 token 更简单（不需要额外的 token 申请接口）
- UUID 碰撞概率极低（1/2^122），可忽略

### ADR-003：乐观锁防并发冲突

**决策**：`exception_tickets.version` 字段 + `UPDATE WHERE version = :current`，冲突时返回 409。

**理由**：
- 审批操作频率低，乐观锁比悲观锁（SELECT FOR UPDATE）更适合
- 不阻塞其他读操作
- 前端收到 409 后提示用户刷新重试，用户体验可接受
